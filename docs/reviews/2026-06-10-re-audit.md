# Re-Audit: Paperless Archivist — 2026-06-10 (post-v1.12.0)

**Scope:** A second pass after the v1.12.0 audit-remediation release. Four review agents read the 29 remediation commits (`5ce1d86`..`v1.12.0`) *and* re-read the whole codebase looking for regressions introduced by the fixes, issues the first audit missed, and improvements. Augmented by a read-only inspection of the running production deployment (the image already includes the v1.12.0 fixes — migration 0034 ran live on startup).

**Headline:** No new criticals in the v1.12.0 changes themselves — most of the 29 commits held up under adversarial review. But the runtime inspection surfaced **one pre-existing critical bug** (provider cooldown keyed wrong) that v1.12.0 made visible, **one active operational gap** (the metrics scrape token was not set when the token-gating change deployed), and **several regressions in the new code** (an over-eager input clamp, an OCR cap that the image path bypasses, a retention/verify key mismatch). Full list tracked as #279–#297.

---

## Active in production right now

**Provider cooldown stored under a hardcoded key, looked up by the configured name → the worker hammers an exhausted provider** — `critical` → #279
`crates/archivist-ai/src/lib.rs:674,717`. The Ollama client's chat/vision quota path passes a literal `"ollama"` to `check_quota_then_take_body` (the other clients correctly pass `self.provider_name`; the field's doc-comment explicitly warns against the hardcode). The cooldown row is written for `"ollama"` but `active_cooldown_for_stage` queries by the configured stage-provider name, which for the cloud Ollama preset is `ollama-cloud`. The lookup never matches, the cooldown short-circuit fails open, and every newly claimed job makes two doomed LLM calls and parks itself — observed flooding the worker logs. Pre-existing (not from v1.12.0); surfaced now because the cloud provider is the one hitting its weekly limit. Two-line fix.

**Metrics scrape token not configured for the deployment** — `high` → #281
The v1.12.0 change that put `/metrics` behind a bearer token shipped, but the token was not added to the deployment environment, so the endpoint returns `503` and metric scraping is down until it is set. Operator action (set the token in the deploy secret + scrape config). Also: config validation accepts an empty token/webhook secret — should be rejected at startup.

**Quota signals swallowed by the doc-type pre-pass and consensus check** — `high` → #280
`crates/archivist-worker/src/main.rs:2026-2036, 2800-2811`. A `QuotaExhausted` from the pre-pass becomes "category Other" and the job proceeds to a second doomed call; the consensus path treats a quota error as "no opinion" and never records a cooldown. Amplifies #279.

## OCR memory & v1.12.0 regressions — `high`/`medium`

- **OCR image fast-path bypasses the render caps** (#282, `high`) — `crates/archivist-ocr/src/lib.rs:44-53`. The v1.12.0 OCR memory caps only fenced the PDF render path; a large image input is returned uncapped, so a 250 MB image expands to roughly 1.7 GB in flight for a single job → OOM on a 1 GiB worker.
- **OCR caps exceed the worker memory limit** (#283, `high`) — worst-case per job ≈ 684 MB (download + render + clones + base64); two concurrent jobs exceed the 1 GiB limit. Caps need to be calibrated to the limit/concurrency, or the limit raised; structural wins (drop the original before the page loop, encode base64 once) help.
- **`clampInteger` clamps on every keystroke** (#284, `high`) — `frontend/src/settings/sections/SecuritySection.tsx` + `helpers.ts`. The v1.12.0 fix makes multi-digit retention/TTL values untypeable (first digit clamps to the minimum) and was only applied to one settings section; the others still save `0` on clear. Fix: raw draft + clamp on blur, applied everywhere.
- **Audit retention deletes by `created_at`, the chain verifies by `chain_position`** (#285, `medium`) — `crates/archivist-db/src/lib.rs:7388` vs `:7239`. Under the clock skew the chain-position change exists to tolerate, a `created_at`-prefix delete can punch a mid-chain hole and raise a false integrity alarm. Delete by chain-position prefix instead.
- **Chat `sendMessage` bypasses the new session-switch guard** (#286, `medium`) and **stale "Unauthorized" banner after re-login** (#287, `medium`) — incomplete corners of the v1.12.0 frontend fixes.

## Security & auth — `medium`/`low`

- **Missing HSTS / CSP / Permissions-Policy** (#288) — the app sets three response headers but no transport-security or content-security policy, contrary to its own `SECURITY_DESIGN.md`. A recommended, build-tested header set is in the issue (`script-src 'self'` does not break the SPA). Add in the existing header layer, HSTS gated on `cookie_secure`.
- **OIDC roles are additive-only** (#289) — removing a user from the admin allowlist does not de-provision their stored roles on the next login.
- **`public_url` is not scheme-validated** (#290) — a settings admin can plant a `javascript:` URL that renders as an executable link for lower-privileged users (the recommended CSP also blocks it).
- **Auth-hardening collection** (#291) — username-link trust vs the email-verified gate, a login-lockout timing oracle, no body cap on the auth routes, and internal error text returned from the public OIDC callback.

---

## Worker / AI robustness — `medium`/`low`

- **`Retry-After` can't shorten the 24 h cooldown + the quota-signal substring is too broad** (#292) — a 60 s throttle parks a provider for a day.
- **Per-provider `num_ctx` preset (4096) overrides the GGML-crash fix** (#293) — vision can run in the crash regime / prompts get truncated.
- **Ollama 4xx errors are untyped** (#294) — a 401/404 is misclassified transient and burns the retry budget.
- **Robustness collection** (#295) — cooldown-release failure sacrifices the document, the doc-type classifier prompt lacks fence-neutralization, full-content lowercasing ×3 and a large hash on the async runtime, the stale-`applying` sweep keys off a non-restamped timestamp, and the numeric-redaction exemption is broader than intended.

## Frontend & ops — `low`

- **Frontend cleanup collection** (#296) — one-time token never cleared from the DOM, alert navigation to a non-renderable tab, missing double-click guards on review approve/reject, a memo that drops title changes, a "Saved" message that never clears, a logout unhandled rejection, a broken label association, and the now-dead `openapi-fetch` dependency.
- **Ops collection** (#297) — align the worker termination grace period with the in-repo manifest, set a `sizeLimit` on the OCR `/tmp` volume and add a pixel cap to the renderer (disk-fill), and finish localizing the Prompts page.

---

## What held up well

The running pods have an exemplary security context (non-root, read-only root filesystem, all capabilities dropped, seccomp `RuntimeDefault`); there is no XSS sink in the SPA; the auth model, CSRF double-submit, SSRF-on-save validation, and the review double-apply fence are sound. Most of the v1.12.0 remediation survived adversarial review unchanged — the regressions found are at the edges (an over-eager clamp, one uncapped code path, a delete/verify key mismatch) rather than in the core of the fixes.

## Method

Four review agents (Fable 5), each read-only: (1) a regression review of the full `5ce1d86..HEAD` diff; (2) worker + AI + OCR pipeline (confirmed the cooldown and OCR-memory findings against the running deployment's logs and limits); (3) HTTP API auth + response headers; (4) the React frontend. Findings were verified against the current tree before filing. Already-fixed issues from the first audit were excluded.
